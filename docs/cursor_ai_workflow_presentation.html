Step 1
Write Rules

Rules are markdown files in .cursor/rules/ that get loaded into every AI conversation. They're how you teach the AI your project once and never repeat yourself.

📚

Context Rules

Teach the AI your project structure — Jira project keys, custom fields, platform names, JQL patterns, environment details. It will use this context in every conversation.

Project key: ART
Platforms: AGG → Q3D, ME100 → NCP6-S
Custom fields: Test Steps, Version, Customer
ALWAYS ON
🔒

Safety Rules

Set guardrails for dangerous operations. Require AI to ask before creating bugs, updating tickets, or modifying production data. Prevent batch operations without per-item approval.

NEVER update a Jira ticket without
explicit user approval first.
GUARDRAIL
▶

Workflow Rules

Define multi-step procedures the AI should follow. For example: "when I say execute a test, fetch the ticket, parse steps, SSH to the device, run commands, report results."

1. Fetch ticket → 2. Parse steps
3. SSH to device → 4. Run commands
5. Build results → 6. Update Jira
WORKFLOW
📋

Template Rules

Enforce consistent output formats. Define templates for bug reports, test results, or documentation so every AI-generated artifact follows the same structure your team expects.

**Title:** <short description>
**Steps to Reproduce:** 1. 2. 3.
**Actual Results:** <with CLI output>

06 / 14
Inside a Rule File
What a Rule
Actually Looks Like

A rule is just a markdown file with a YAML header. The alwaysApply: true flag means it loads automatically in every conversation. Set it to false for rules the AI only uses when relevant.

bug-summary-format.mdc
---
description: Required format for bug reports
alwaysApply: true
---

# Bug Summary Format

When reporting a bug, ALWAYS use this template:

**Title:** <short, descriptive>
**Issue Summary:** <the defect>
**Environment:** <host, version, platform>
**Expected:** <what should happen>
**Actual:** <what happened + CLI output>
**Steps to Reproduce:** 1. 2. 3.

## Rules
- Never omit any section
- Include hostname or IP
- Steps must be numbered
- Include verbatim CLI output
jira-approval.mdc
---
description: Jira changes require approval
alwaysApply: true
---

# Jira Changes Require Approval

Before creating any Jira bug ticket,
you MUST:

1. Present the bug summary to the user
2. Ask for explicit permission
3. Only create after user confirms

NEVER update, modify, or edit any
existing Jira ticket without explicit
user approval first. This includes:

- Updating fields or status
- Adding comments
- Linking or unlinking issues
- Changing assignees

Step 2
Connect Your Tools

MCP (Model Context Protocol) lets the AI talk directly to external systems. Configure servers in .cursor/mcp.json and the AI can query Jira, SSH to devices, post to Slack — without you copy-pasting anything.

Jira / Confluence

AI reads tickets, creates bugs, updates fields, adds comments — all through natural language

Network / Lab Devices

Discovers topology, resolves device names, runs CLI commands over SSH on real hardware

GitHub

Reads repos, searches code, checks PRs — connects code to your test workflow

Browser

Navigates web UIs, clicks, fills forms, takes screenshots for UI verification

Slack

Reads channels, posts updates, searches conversations from within the IDE

Custom APIs

Any REST/SSE endpoint can become an MCP server — extend AI to any internal tool

HOW IT WORKS

You say: "Check the status of SW-248225"

AI does:

1. jira_get_issue("SW-248225")
2. Reads status, assignee, comments
3. Returns: "SW-248225 is In Progress, assigned to Dan, last comment says fix is in branch v25.4-hotfix"

You say: "Run 'show cfm status' on the AGG device"

AI does:

1. get_device_management_interfaces("AGG")
2. run_show_command(device, "show cfm status")
3. Returns the parsed CLI output directly in the chat

Real Example
Test Execution Workflow

This is what happens when you say: "Run the test for ART-8025 on device 100.64.6.171"

1

AI fetches the Jira ticket via MCP

Reads summary, description, and test steps from custom field. No copy-pasting needed.

atlassian_jira_get_issue(issueKey: "ART-8025")
2

Parses test steps and shows you what it will run

Extracts numbered steps, CLI commands, and expected results from the ticket. Asks you to confirm before executing.

3

SSHs to the device and runs each command

Connects via MCP or paramiko, captures software version, then executes each test step sequentially.

show cfm maintenance-association default eth-dm status | no-more
4

Builds PASS/FAIL results with CLI evidence

Compares actual output to expected results. Formats everything in Jira wiki markup with green checks and red X marks.

5

Shows you the results — asks permission before updating Jira

You review the formatted results. Only after you approve, the AI appends Test Results to the ticket description or adds a comment.

Step 3
Plan Before You Build

For complex tasks, ask the AI to create a plan first. Plans live in .cursor/plans/ with tracked TODOs. The AI thinks through the approach, you review it, then it executes step by step.

01

Describe the Goal

"I need to test Y.1731 SLM with different PCP values across all 8 priority levels"

02

AI Explores & Plans

AI reads the Jira ticket, checks existing tests for patterns, creates a plan with phases: setup, configure, execute, verify, cleanup

03

You Review & Adjust

Read the plan, add constraints ("use paramiko not pexpect"), remove unnecessary steps, approve the approach

04

AI Executes with TODOs

AI works through each TODO — writes code, runs tests, marks items complete. You see progress in real time.

REAL PLAN EXAMPLE
name: CLI Test Gap Coverage
todos:
  [x] inform-test-results disabled check
  [x] TAB completion for all DM/SLM commands
  [x] PCP boundary values (0 and 7)
  [x] Dependency deletion guard
  [x] Show command coverage
  [x] Operational state validation
  [x] Historic results verification
  [x] System event testing

Step 4
How to Talk to the AI

DO THIS

Be specific: "Write a test for SW-236668 that verifies ETH-DM delete syntax on NCP6-S"

Reference existing code: "Follow the same pattern as y1731_cli_tab_test.py"

Give constraints: "Use paramiko, output results as JSON, add --host argument"

Ask it to plan first: "Create a plan for this before writing code"

Review & iterate: "The SSH timeout is too short, change it to 30s"

Let it use tools: "Check ART-8025 in Jira and tell me the test steps"

AVOID THIS

Vague requests: "Write me a test" — for what? on what platform?

No context: Starting fresh without rules means re-explaining everything

Accepting blindly: Always review generated code — AI can make mistakes

Skipping plans for big tasks: Going straight to code on complex features leads to rework

No guardrails: Without safety rules, AI might update Jira tickets or push code without asking

Manual copy-paste: If you're copying from Jira into the prompt, use MCP instead

Advanced
Power User Tricks
⚙

Layer Your Rules

Use alwaysApply: true for rules that apply everywhere (project context, safety). Use alwaysApply: false for task-specific workflows that the AI activates only when relevant.

always on: jira-mcp, approval, bug-format
on demand: execute-testing-task
🔍

Feed It Reference Code

Keep source code, YANG models, and MIBs in a reference/ folder. The AI reads these for correctness — it understands the actual implementation, not just your description of it.

reference/CfmSessionSLM.cpp
reference/dn-performance-monitoring.yang
reference/DRIVENETS-CFM-MIB.mib
🔧

MCP from Python Scripts

MCP isn't only for Cursor. You can call MCP tools from standalone Python scripts for CI, cron jobs, or bug reproduction — same device discovery and CLI execution outside the IDE.

from lib.nm_call import mcp_call
mcp_call("run_show_command",
  {"device": "AGG", "cmd": "show cfm"})
📄

Write Runbooks as AI Procedures

Structure docs with numbered steps, CLI placeholders, and pass criteria. The AI can follow runbooks step-by-step, filling in device names and checking results automatically.

1. SSH to <DEVICE>
2. Run: show cfm <MD> <MA> <MEP>
3. Verify: status = "Active"
📈

Structured JSON Output

Have test scripts write results as JSON, not just logs. The AI can read, compare, and analyze structured results across runs — much harder with raw text output.

{"test": "SLM PCP-7", "result": "PASS",
 "near_end_loss": 0, "far_end_loss": 0}
🔌

MCP Feature Flags

Control which tools are available per MCP server using HTTP headers. Enable/disable Jira write, Slack, GitHub, or RAG independently without changing the server.

X-JIRA-TOOLS: enabled,write,mgmt
X-SLACK-TOOLS: enabled,write
X-GITHUB-TOOLS: enabled
